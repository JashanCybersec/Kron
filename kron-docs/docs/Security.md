# KRON — Security Architecture

**Version:** 1.0

---

## Security Philosophy

KRON is a security product. Its own security posture must be held to a higher standard than the systems it monitors. Every design decision follows three principles:

1. **Assume breach** — design as if an attacker already has a foothold somewhere
2. **Defense in depth** — no single control is trusted completely; multiple independent layers for every critical protection
3. **Minimal blast radius** — if any component is compromised, limit what the attacker can reach, read, or do

---

## Authentication

### User Authentication
- **Password hashing:** Argon2id (memory: 64MB, iterations: 3, parallelism: 4)
- **MFA:** TOTP (RFC 6238) required for all roles. Hardware keys (FIDO2/WebAuthn) supported in Enterprise
- **Session tokens:** signed JWT (RS256), 8-hour expiry, non-renewable (re-auth required)
- **Brute-force protection:** 5 failed attempts → 15-minute lockout → page admin after 3 lockouts
- **Login anomaly detection:** KRON applies its own UEBA rules to KRON login events

### Service Authentication (Standard/Enterprise)
- **mTLS:** Every inter-service call uses mutual TLS
- **Standard tier:** Self-signed CA managed by KRON, certs auto-rotated every 90 days
- **Enterprise tier:** SPIFFE/SPIRE issued X.509 SVIDs, 4-hour TTL, auto-rotated
- **No API key for internal services** — certificate identity only

### Agent Authentication
- Agents authenticate to the collection endpoint using a per-agent client certificate
- Certificates issued by KRON on agent registration, 30-day TTL
- Agent registration requires a one-time token generated by admin
- Revocation: agent cert revoked immediately on deregistration or compromise signal

## Authorization

### RBAC Model

| Role | Read Events | Write Rules | Execute SOAR | Admin |
|---|:---:|:---:|:---:|:---:|
| viewer | ✓ | — | — | — |
| analyst | ✓ | ✓ | — | — |
| responder | ✓ | ✓ | ✓ | — |
| admin | ✓ | ✓ | ✓ | ✓ |
| mssp_admin | own tenants only | own tenants only | own tenants only | own tenants only |

### Tenant Isolation (4 independent gates)

**Gate 1 — JWT claim:** Every token carries `tenant_id`. Invalid or missing → 401.

**Gate 2 — Query rewrite middleware:** Every SQL query has `AND tenant_id = $1` injected by Rust middleware before execution. Raw queries from clients are never passed through directly.

**Gate 3 — ClickHouse row policy:**
```sql
CREATE ROW POLICY tenant_isolation ON events
    FOR SELECT USING tenant_id = currentSetting('kron.tenant_id')
    TO kron_api_user;
```
Even a direct database connection cannot bypass this.

**Gate 4 — Continuous canary test:** Every 5 minutes, an automated test executes a cross-tenant query and asserts zero rows returned. Alert fires if any rows are returned.

### Admin Action Controls (Enterprise)

All destructive admin actions require 4-eyes approval:
- Alert suppression
- Rule disable/delete
- Log purge (outside TTL)
- Tenant deletion
- HSM key operation

Implementation: action creates a pending request → second admin approves via separate session → action executes → both events written to audit log. No self-approval possible.

## Encryption

### In Transit
- **TLS 1.3 minimum** everywhere (rustls, no OpenSSL dependency)
- **TLS 1.2** supported for legacy agent compatibility only
- **Cipher suites:** TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256
- **Perfect forward secrecy** on all connections
- **HSTS** enforced on web UI (max-age=31536000, includeSubDomains)

### At Rest

**Nano tier:**
```
systemd-creds (OS keyring / TPM-backed)
  └─ AES-256-GCM per file
       └─ KRON config, agent tokens, API keys
```

**Standard tier:**
```
Vault (software KMS)
  └─ Master key (derived from machine identity + admin passphrase)
       └─ Per-tenant DEK (AES-256-GCM)
            └─ ClickHouse cold storage (Parquet files)
            └─ Sensitive config values
```

**Enterprise tier:**
```
HSM (PKCS#11)
  └─ Master KEK (never leaves HSM)
       └─ Per-tenant KEK (AES-256)
            └─ Per-partition DEK (AES-256-GCM, rotated monthly)
                 └─ ClickHouse data (all tiers)
                 └─ Parquet cold store
                 └─ Audit log
```

Key rotation:
- DEKs rotated monthly (automated)
- Tenant KEKs rotated annually (automated)
- Master KEK rotation: manual, dual-person control, fire drill required

### Merkle-Chained Audit Log

Every row in `audit_log` is cryptographically linked:

```
row_hash = SHA256(audit_id || ts || actor_id || action || resource_id || result || prev_hash)
```

The `prev_hash` of row N is the `row_hash` of row N-1. Chain break = tampered row. Verified on demand and on every compliance report generation.

Audit log rows are append-only. No UPDATE or DELETE allowed at the ClickHouse user permission level.

## Network Security

### KRON Server Hardening

```bash
# Firewall rules — only expose what is necessary
ufw default deny incoming
ufw default allow outgoing
ufw allow 443/tcp    # KRON web UI and API (TLS)
ufw allow 8883/tcp   # KRON agent ingestion (mTLS)
ufw allow 22/tcp     # SSH (key-only, from bastion only)
# All other ports blocked externally — ClickHouse, Redpanda, Vault 
# accessed only via loopback or Cilium mesh
```

### Cilium Network Policy (Enterprise)

Deny-by-default. Only explicitly declared service pairs can communicate:

```yaml
# Example: only stream-processor can write to ClickHouse
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
spec:
  endpointSelector:
    matchLabels:
      app: clickhouse
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: stream-processor
    toPorts:
    - ports:
      - port: "9000"
        protocol: TCP
```

No pod can reach ClickHouse directly except the stream-processor and query-api. Stream-processor cannot reach the web UI. SOAR engine cannot reach ClickHouse directly.

### Agent Transport Security

- Agent → KRON collection endpoint: TLS 1.3, client certificate auth
- Connection: long-lived gRPC stream (reduces TCP handshake overhead)
- Backpressure signal propagated from server to agent (agent buffers if server is slow)
- Agent does not store events in plaintext on disk — disk buffer is encrypted with agent-local key

## Vulnerability Management

### Dependency Scanning
- `cargo audit` runs on every commit — blocks merge if HIGH/CRITICAL CVEs
- `cargo deny` enforces license allowlist and version constraints
- Dependabot auto-raises PRs for dependency updates weekly

### SAST
- `clippy` with security-relevant lints enabled
- `miri` for undefined behaviour detection on critical paths
- Custom lint rules for: raw SQL construction, unvalidated input, unsafe blocks

### Penetration Testing
- Internal penetration test before v1 GA (founder or contracted)
- Annual third-party pentest from CERT-In empanelled auditor
- Bug bounty programme post-launch (in-scope: KRON itself, not monitored orgs)

### Incident Response for KRON Vulnerabilities
- CVSS 9.0+: patch within 48 hours, notify all customers within 72 hours
- CVSS 7.0–8.9: patch within 7 days
- CVSS <7.0: next scheduled release
- Security advisory published for all CVEs affecting production

## Supply Chain Security

### Build Pipeline
- All builds run in GitHub Actions ephemeral runners
- Reproducible builds (Rust lockfile committed, verified in CI)
- Container images built from pinned base images (`ubuntu:22.04@sha256:...`)
- SBOM generated for every release (CycloneDX format)
- All release artifacts signed with Sigstore cosign

### Dependency Policy
- No `unsafe` Rust in first-party code without documented justification
- All `unsafe` blocks require code review by two engineers
- Third-party crates: max 90 days behind latest version for security crates
- No crates with unmaintained status (last commit >2 years) in critical path

## Data Privacy

### What KRON Collects
KRON only collects security-relevant telemetry. It does not:
- Capture the content of encrypted communications
- Log personal email body content
- Store passwords or credentials (it detects credential theft, not credentials themselves)
- Transmit any customer data to KRON Ltd infrastructure

### Data Minimization
Raw event collection can be filtered at the agent level. Customers can exclude specific paths, usernames, or source types from collection.

### DPDP Act Compliance (KRON itself)
- KRON collects only: name, email, phone for customer accounts
- Data stored in India (customer's own infrastructure — KRON doesn't host it)
- Customer can export and delete their own data at any time
- Data Processing Agreement (DPA) available for enterprise customers

## Security Monitoring of KRON Itself

KRON ingests its own logs and applies detection rules against itself. Additional KRON-specific rules:

- Unusual admin login time or location
- Admin action rate anomaly
- Merkle chain break (immediate P1)
- Cross-tenant query returning rows (immediate P1, escalates to founder)
- ClickHouse direct access from non-whitelisted service
- Vault secret access outside normal patterns
- Agent mass-deregistration (potential agent fleet takeover)
- API rate limit bypass attempt

KRON meta-alerts route to a separate notification channel that no KRON admin can suppress.
